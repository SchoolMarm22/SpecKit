# What We're Looking For

We're not building another CRUD app with a React template. Linear is a craft product — every interaction should feel instant, every animation should run at 60fps, and keyboard navigation should be faster than reaching for your mouse. If you've never obsessed over frame timing or debated whether a 16ms delay is perceptible, this probably isn't the role for you.

We ship fast but we don't ship garbage. TypeScript isn't optional here — it's how we move quickly without breaking things. We're looking for someone who sees performance as a core product feature, not a nice-to-have that gets deprioritized for the next sprint.

You'll be working directly on the core Linear experience that thousands of engineering teams use daily. No enterprise customizations, no feature flags for different customer tiers — just one really good product that we make better every week.

## Must-Haves

- Shipped complex interactive web applications where performance actually mattered (can explain specific optimizations you made and why)
- Deep TypeScript experience — not just "I add types sometimes" but actually leveraging the type system to catch bugs and improve DX
- Strong opinions about animation and micro-interactions in web apps, with examples of implementation
- Experience with state management in large frontend applications (Redux, Zustand, or similar)
- Can articulate why you chose specific technical approaches and what tradeoffs you made

## Strong Signals

- Has built keyboard-first interfaces or complex keyboard navigation systems
- Experience with canvas, WebGL, or other performance-sensitive web technologies
- Contributed to or maintained developer tools, CLIs, or other products used by engineers
- Can demo smooth 60fps animations in a web browser and explain how they work
- Has strong opinions about TypeScript patterns and can defend them

## Anti-Patterns

- Treats TypeScript as "JavaScript with some type annotations"
- Has never measured or optimized render performance in a React app
- Dismisses animation and polish as "not real engineering work"
- Only experience is with heavily abstracted corporate codebases where performance wasn't a concern
- Can't explain technical decisions without falling back to "that's just how we always did it"

## Interview Focus

- Live coding session focused on TypeScript patterns and performance considerations
- Technical deep-dive into a complex interactive feature they've built
- Discussion of animation/interaction design decisions and their implementation
- How they approach debugging performance issues in production web apps
- Their philosophy on balancing shipping speed with code quality
