# What We're Looking For

You'll be building the frameworks that millions of developers use, but they'll never know your name. That's the job. We don't ship features — we ship capabilities. When SwiftUI launched, it wasn't because we rushed to market. It was because we spent years getting the API surface exactly right, knowing that once we ship, we own that API forever.

Small team, massive impact. You'll be the DRI for frameworks that ship to every iOS device. When you make a decision about memory management or threading models, you're making that decision for the entire ecosystem. The weight of that responsibility should energize you, not intimidate you.

We care more about doing it right than doing it first. If that sounds frustrating, this isn't the role for you.

## Must-Haves

- 5+ years building iOS frameworks or SDKs (not apps) with deep understanding of binary compatibility and API evolution
- Expertise in Swift runtime internals and Objective-C interoperability — you understand what happens when Swift calls @objc code
- Experience with performance profiling at the framework level using Instruments, not just fixing app-level performance issues
- Can articulate the tradeoffs between compile-time safety and runtime flexibility in API design
- Track record as a DRI on projects with 6+ month timelines where changing requirements meant rethinking fundamental assumptions

## Strong Signals

- Has designed APIs that other engineers loved using — can walk through specific API design decisions and their reasoning
- Deep iOS platform knowledge: understands how Core Animation, Grand Central Dispatch, or similar system frameworks actually work under the hood
- Experience with static analysis tools or compiler development
- Has worked on backwards compatibility challenges with shipped APIs

## Anti-Patterns

- "Framework experience" that was really just creating internal utility classes for app teams
- Can't explain why certain Apple APIs are designed the way they are (shows lack of systems thinking)
- Wants to ship fast and iterate — that's not how platform engineering works
- Talks about "moving fast and breaking things" or similar Valley mantras that don't align with our craft-first culture
- Has never been the sole owner of something complex and long-term

## Interview Focus

- API design philosophy: what makes a good framework API vs. a good app API
- How they handle the tension between developer ergonomics and system performance
- Concrete examples of framework-level debugging or performance optimization
- Understanding of compilation, linking, and runtime behavior in iOS
- How they approach long-term architectural decisions when requirements will change but the API cannot
