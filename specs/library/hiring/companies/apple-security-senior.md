# What We're Looking For

You'll be the DRI for security domains that ship to hundreds of millions of devices. This isn't enterprise security or compliance theater — you're protecting actual people's most personal data while enabling magical user experiences that feel effortless.

At Apple, security engineering means thinking like an attacker while building like a craftsperson. You'll work across the full stack from kernel-level hardening to cryptographic protocols, often with teams you can't talk to about what you're building until it ships. The person we're looking for has internalized that great security is invisible security.

We move deliberately. A security vulnerability that ships to a billion devices can't be patched with a quick hotfix blog post. You need to think in terms of hardware generations, not sprint cycles.

## Must-Haves

- Shipped cryptographic implementations to production (not just integrated libraries)
- Deep understanding of iOS/macOS security model — Secure Enclave, code signing, sandboxing, etc.
- Experience with low-level security: kernel hardening, secure boot chains, or hardware security modules
- Has been responsible for a security decision that affected millions of users
- Can articulate why certain security tradeoffs were made in shipping Apple products

## Strong Signals

- Published security research or found vulnerabilities in production systems
- Experience with privacy-preserving technologies (differential privacy, homomorphic encryption, secure multiparty computation)
- Has worked on security features that users never see but always benefit from
- Understanding of hardware-software security integration
- Can explain complex security concepts to non-security engineers without losing precision

## Anti-Patterns

- Security experience limited to compliance frameworks or enterprise tools
- Treats security as a checkbox rather than a design philosophy
- Can't explain how their security work enabled user features
- Focuses on theoretical attacks without practical exploitation experience
- Dismissive of usability constraints or thinks friction equals security

## Interview Focus

- Technical depth: can they walk through a complete attack chain and explain mitigations at each layer?
- Product thinking: how do they balance security with user experience?
- Systems understanding: how would they secure a new hardware/software integration?
- Craft mindset: examples of security work they're proud of and why
- Discretion: can they discuss technical details while respecting confidentiality boundaries?
