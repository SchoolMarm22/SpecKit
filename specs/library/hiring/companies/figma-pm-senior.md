# What We're Looking For

We're not building enterprise software for people who hate using it. We're building tools that developers choose to love, the same way designers choose Figma over legacy alternatives. That means you need to feel the difference between good and great developer experience in your bones.

Our developer platform serves millions of plugin developers, widget creators, and API consumers. When we ship something clunky, it doesn't just hurt adoption numbers—it breaks the creative flow that makes Figma magical. You need to understand that our platform IS the product for developers, not a side project with APIs.

The technical complexity here is real: real-time multiplayer, WebGL rendering, C++ performance constraints compiled to WASM. But the product complexity is equally real: how do you expose powerful primitives without overwhelming someone building their first plugin? How do you maintain API stability when the core product moves at startup speed?

## Must-Haves

- Built developer tools or platforms that developers actually adopted (not just used because they had to)
- Can articulate the difference between developer experience and user experience, with specific examples
- Technical enough to have opinions about API design beyond "make it RESTful"
- Has shipped something where performance constraints materially shaped product decisions
- Experience with real-time or multiplayer systems (games, collaborative tools, live data)

## Strong Signals

- Built their own tools or plugins for software they use professionally
- Has strong opinions about documentation that go beyond "write more docs"
- Understands why we chose WebGL over SVG and can explain similar technical tradeoffs
- Experience with developer communities—knows the difference between engagement and advocacy
- Can talk about a time they said no to a feature request and why the developers thanked them for it

## Anti-Patterns

- Treats developers like enterprise buyers (personas, procurement, RFPs)
- API-first thinking without understanding the rendering or performance implications
- Can't explain why real-time collaboration is technically hard
- Views platform work as "enabling the real product team"
- Measures success primarily through adoption metrics without understanding developer sentiment

## Interview Focus

- API design philosophy: show them our plugin API and ask what they'd change and why
- Performance tradeoffs: how do you decide what complexity to expose vs. abstract away?
- Community dynamics: how do you build for power users without alienating newcomers?
- Technical collaboration: how do you work with engineers when you don't write the code but need to understand the constraints?
- Real examples of developer tools they've loved or hated, and what made the difference
