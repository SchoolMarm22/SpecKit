# What We're Looking For

This isn't your typical frontend role. Search operates at a scale where a 0.1% improvement in click-through rate impacts millions of users daily. You'll be writing TypeScript that renders for 8 billion searches per day, where performance optimizations are measured in nanoseconds and A/B tests run on populations larger than most countries.

The Search frontend team owns everything from the search box autocomplete to results rendering, knowledge panels, and search features that haven't been announced yet. We operate in a world where design docs are mandatory, code reviews are comprehensive, and your impact is measured by launch metrics that move at Google scale. If you've only worked at companies where "scale" means thousands of users, this will be a fundamental shift in how you think about engineering.

You'll be working in google3 with our internal web framework stack. External React experience is valuable, but you'll need to prove you can adapt to our internal tools and contribute meaningfully to a codebase with over 2 billion lines of code.

## Must-Haves

- 5+ years shipping production JavaScript at companies with >10M users
- Experience with performance optimization that goes beyond "I used React.memo"—actual profiling, bundle analysis, and measurable improvements
- Strong systems thinking: can design frontend architecture that scales across multiple teams and products
- Proven track record of technical leadership through design docs and code review
- Experience working with large, established codebases (not just greenfield projects)

## Strong Signals

- Has written design docs that influenced technical decisions across multiple teams
- Experience with web performance at scale: Core Web Vitals, rendering optimization, progressive loading
- Previous experience at companies with rigorous code review culture (Facebook, Microsoft, Netflix-scale engineering)
- Can articulate specific performance improvements they've made and how they measured impact
- Understanding of accessibility standards and internationalization challenges

## Anti-Patterns

- "Frontend lead" experience that was really just React component work
- Can't discuss performance beyond framework-level optimizations
- Doesn't understand the business impact of their technical decisions
- Treats code review as a checkbox rather than collaborative design process
- Never worked on products where backwards compatibility and gradual rollouts are critical

## Interview Focus

- System design: How would you architect a feature that needs to scale to billions of daily users?
- Performance deep-dive: Walk through a specific optimization you made, including measurement methodology
- Design doc discussion: How do you approach technical design when multiple teams depend on your API?
- Code quality: What makes code reviewable and maintainable in a 2-billion-line codebase?
- Impact measurement: How do you know if a frontend change actually improved the user experience?
