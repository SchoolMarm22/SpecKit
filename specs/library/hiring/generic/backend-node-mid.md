# What We're Looking For

I want someone who's moved beyond just implementing features and has started thinking about the system. Mid-level means you can take a project from requirements to deployment without constant oversight, but you're still growing into architectural thinking.

The sweet spot is 3-5 years of real Node.js experience. Not someone who's been writing JavaScript for 5 years and touched Node once, but someone who understands the event loop, has debugged memory leaks, and knows why `await` in a loop might be a bad idea.

We're looking for someone who can own a service end-to-end but doesn't need to design the entire system architecture yet. You should be the person junior engineers come to with questions, but you're still learning from the seniors about scaling and system design.

## Must-Haves

- Built and deployed Node.js APIs in production (Express, Fastify, or similar)
- Experience with both SQL and NoSQL databases, can articulate when to use each
- Written and maintained automated tests (unit, integration, and basic load testing)
- Comfortable with Docker and basic AWS/GCP services (S3, RDS, EC2 basics)
- Has debugged performance issues in production Node.js applications

## Strong Signals

- Contributes to technical discussions in code review with thoughtful feedback
- Has opinions about API design patterns and can defend REST vs GraphQL tradeoffs
- Experience with message queues or event-driven architectures (Redis, RabbitMQ, SQS)
- Understands monitoring and observability beyond just logging errors
- Has mentored or pair-programmed with junior developers

## Anti-Patterns

- Can only work with one framework and gets lost without it
- Writes tests only when forced to, doesn't see the value
- Never asks questions or always asks the same questions repeatedly
- Treats all performance problems as "just add more servers"
- Has strong opinions about tools but weak understanding of underlying concepts

## Interview Focus

- Code organization: how they structure a new service from scratch
- Debugging stories: walk through a real production issue they solved
- Testing philosophy: what they test, what they don't, and why
- Database choices: when they'd pick Postgres vs MongoDB vs Redis
- Basic system design: not architecture, but service-level decisions
