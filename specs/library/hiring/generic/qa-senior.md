# What We're Looking For

I'm looking for a QA engineer who thinks like a software engineer first and a tester second. Someone who's moved beyond "click through the UI and file bugs" to building systems that prevent bugs from reaching production in the first place.

The best QA engineers I've worked with are paranoid optimists — they assume everything will break, but they build elegant solutions to catch it early. They're the ones who see a flaky test and think "this is a systems problem" not "let's just retry it." They write test automation that other engineers actually want to use and maintain.

## Must-Haves

- Built and maintained test automation frameworks (not just written tests)
- Owned end-to-end test strategy for a product or major feature
- Experience with API testing beyond just happy path scenarios
- Can write performant, maintainable test code that other engineers respect
- Has improved team velocity through better testing practices (can quantify the impact)

## Strong Signals

- Strong opinions about test pyramid and can explain when to violate it
- Experience with contract testing, chaos engineering, or property-based testing
- Has mentored engineers on testing practices and seen them improve
- Built custom tooling or plugins to solve testing problems
- Can talk intelligently about tradeoffs between test coverage and execution time

## Anti-Patterns

- Treats automation as "record and playbook" without understanding the underlying code
- Can't explain why they chose Selenium over Playwright (or vice versa) beyond "it's what we used"
- Views their role as "quality gatekeeper" rather than "quality enabler"
- Has never had to debug a flaky test or doesn't understand why flaky tests are poison
- Only experience is manual testing or junior-level automation work

## Interview Focus

- Testing philosophy: when do you write unit vs integration vs e2e tests, and why?
- Framework design: how would you build test automation for a new product from scratch?
- Flaky test stories: how they've diagnosed and fixed non-deterministic test failures
- Cross-team collaboration: how they've influenced engineering practices beyond just QA
- Technical depth: can they read and understand the application code they're testing?
