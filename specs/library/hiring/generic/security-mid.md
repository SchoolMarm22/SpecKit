# What We're Looking For

I need someone who's moved beyond just running vulnerability scanners and writing reports. You should have enough battle scars to know that security isn't about perfect compliance — it's about understanding business risk and making pragmatic tradeoffs.

You don't need to be the person designing our entire security architecture yet, but you should be able to own a security domain end-to-end. Whether that's our SIEM implementation, our secrets management strategy, or our application security program, I want someone who can take a problem, research solutions, and drive it to completion without constant hand-holding.

## Must-Haves

- Hands-on experience with at least one major security domain (AppSec, Infrastructure Security, or Detection Engineering)
- Built or maintained security tooling that other engineers actually use
- Participated in real incident response — not tabletops, but actual breaches or security events
- Can code well enough to automate repetitive security tasks (Python, Go, or similar)
- Experience working with engineering teams to fix security issues (not just filing tickets)

## Strong Signals

- Has opinions about security tool sprawl and can explain why consolidation matters
- Understands the business impact of security decisions, not just technical risk
- Can explain why they chose one security control over another in a previous role
- Has contributed to security standards or runbooks that outlived their tenure
- Experience with cloud security (AWS/GCP/Azure) beyond just enabling GuardDuty

## Anti-Patterns

- Security experience limited to compliance audits or penetration testing reports
- Treats developers as adversaries rather than partners
- Can't explain security decisions in business terms
- Only comfortable with commercial tools — never built or customized anything
- Focuses exclusively on preventing attacks without considering detection and response

## Interview Focus

- Walk through a security project they owned from problem identification to solution deployment
- How they've balanced security requirements with developer productivity
- Their approach to threat modeling for a system they've worked on
- Examples of security automation they've built and why it mattered
- How they stay current with the threat landscape and evaluate new security technologies
