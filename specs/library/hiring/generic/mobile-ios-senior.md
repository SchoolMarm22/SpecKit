# What We're Looking For

I need someone who's shipped iOS apps that real people use daily. Not demo apps, not prototypes — production apps with IAP, push notifications, background processing, and all the messy edge cases that come with the territory.

Most importantly, I want someone who understands that mobile engineering isn't just frontend work. You're dealing with offline states, battery constraints, memory pressure, and App Store politics. The best mobile engineers think like systems engineers who happen to work on phones.

## Must-Haves

- Shipped 3+ iOS apps to the App Store, at least one with >10k active users
- Owns Swift and UIKit deeply — can debug auto layout issues without Stack Overflow
- Experience with iOS architecture patterns (MVVM, Coordinator, etc.) and can articulate when to use which
- Has dealt with App Store rejections and knows how to navigate review guidelines
- Understands Core Data or equivalent persistence layer at a deep level

## Strong Signals

- Built custom UI components from scratch (not just configuring system components)
- Experience with SwiftUI adoption strategy in a legacy UIKit codebase
- Has mentored junior iOS engineers through their first production app
- Comfortable with CI/CD for mobile (fastlane, TestFlight automation, etc.)
- Can speak intelligently about app performance profiling and optimization

## Anti-Patterns

- "iOS experience" that's really just React Native or Flutter
- Can't explain memory management in iOS beyond "ARC handles it"
- Has only worked on apps that were essentially web views
- Dismisses Android or backend concerns as "not my problem"
- Can't articulate the tradeoffs of different architecture patterns they've used

## Interview Focus

- App Store submission war stories — what went wrong, how they fixed it
- Technical decision they made that they'd do differently now
- How they approach code reviews for mobile-specific concerns (performance, battery, UX)
- Cross-platform collaboration — how they work with Android engineers and backend teams
