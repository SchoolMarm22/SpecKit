# What We're Looking For

I need someone who thinks in systems, not just services. At staff level, you're not just writing code — you're shaping how we write code. You should have strong opinions about Python that go beyond PEP 8, and be able to articulate why FastAPI vs Django vs Flask matters for different problem domains.

The biggest difference between senior and staff is influence radius. A senior engineer makes their team better. A staff engineer makes the entire backend organization better. I'm looking for someone who's been the technical voice in architecture decisions that affected multiple teams, and who has the battle scars to prove it.

You'll own our Python standards, drive major architectural decisions, and be the person other engineers come to when they're stuck on hard problems. This isn't a people management role, but it is a leadership role.

## Must-Haves

- 8+ years Python in production, with at least 3 years architecting systems that other teams depend on
- Has designed and implemented a major system migration (database, framework, architecture) that affected multiple services
- Deep understanding of Python performance characteristics — when asyncio helps, when it hurts, memory profiling, GIL implications
- Experience with distributed systems patterns: circuit breakers, bulkheads, eventual consistency, saga patterns
- Has been the technical decision maker for choosing between competing solutions at an organizational level

## Strong Signals

- Contributed to major Python libraries or frameworks (Django, FastAPI, SQLAlchemy, etc.)
- Has strong opinions about API design and can defend them with real examples
- Experience building developer tooling or internal platforms that other engineers use
- Can speak to database design decisions beyond just "we normalized the schema"
- Has mentored engineers who are now senior+ themselves

## Anti-Patterns

- Can't explain when to use async Python vs sync Python beyond "async is faster"
- Has never been responsible for a technical decision that went wrong at scale
- Treats all performance problems as database problems or all database problems as query problems
- Describes architecture decisions in terms of what was trendy rather than what solved the business problem
- Has only worked in environments where infrastructure/devops was completely abstracted away

## Interview Focus

- System design scenarios that require making tradeoffs between consistency, availability, and partition tolerance
- Python-specific architecture decisions: when to use microservices vs monolith, async patterns, package structure at scale
- Technical leadership stories: how they influenced technical direction across teams, handled technical disagreements, drove adoption of new patterns
- Code review philosophy and examples of feedback they've given that changed how someone thinks about code
- War stories about debugging production issues in distributed systems
