# What We're Looking For

I need someone who's been in the trenches during a real security incident — not just ran a vulnerability scanner and called it a day. The difference between a mid-level and senior security engineer isn't just technical depth, it's judgment under pressure and the ability to make architectural decisions that don't break the business.

We're looking for someone who can think like an attacker but build like an engineer. Too many security folks can find problems but can't propose practical solutions. I want someone who's built security tooling, not just configured it, and who understands that perfect security that nobody uses is worthless security.

## Must-Haves

- Led incident response for a real security breach (not a drill)
- Built or significantly architected security tooling that other engineers actually use
- Deep understanding of at least two domains: application security, infrastructure security, or detection engineering
- Can code in Python/Go/Java well enough to review security implications of production code
- Experience implementing security controls that scale (auth systems, secrets management, etc.)

## Strong Signals

- Has strong opinions about threat modeling that go beyond STRIDE frameworks
- Built security automation that reduced manual work by >50%
- Successfully pushed back on product requirements for legitimate security reasons
- Experience mentoring engineers on secure coding practices
- Can explain complex security concepts to non-technical stakeholders

## Anti-Patterns

- Security experience that was just running Nessus scans and filing Jira tickets
- Treats security as a checkbox exercise rather than risk management
- Can't code beyond basic scripting (this isn't 2010)
- "We should block everything" mentality without understanding business context
- Has never had to justify security spend to leadership

## Interview Focus

- Incident response: what was the attack vector, how did they contain it, what were the lessons learned
- Technical architecture: how they've designed security controls that developers don't hate
- Risk assessment: how they prioritize vulnerabilities when you can't fix everything
- Cross-team influence: examples of getting non-security teams to adopt better practices
- Tool building: something they built that made security easier for others
