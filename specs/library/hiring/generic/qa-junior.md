# What We're Looking For

I want someone who breaks things naturally — not maliciously, but with genuine curiosity about edge cases and failure modes. The best junior QA engineers I've hired were the people who, as kids, took apart toys to see how they worked and somehow always found the one way to make them stop working entirely.

This isn't about having years of testing experience. It's about having the right mindset: methodical, curious, and comfortable with ambiguity. I'd rather hire someone who's never used Selenium but asks great questions about user workflows than someone who knows every testing framework but can't think beyond happy path scenarios.

## Must-Haves

- Can write basic test cases from user requirements (doesn't need to be perfect, but shows they can think through user flows)
- Comfortable with command line basics and can navigate file systems, run scripts
- Has broken something in a methodical way — could be testing a friend's app, finding bugs in open source projects, or even non-software testing experience
- Can articulate the difference between functional and non-functional testing (even if they haven't done both professionally)

## Strong Signals

- Experience with any automation tool, even if it's just browser record/playback
- Has used APIs directly (Postman, curl, or wrote simple scripts to hit endpoints)
- Asks clarifying questions about requirements instead of making assumptions
- Shows genuine interest in how software fails and why

## Anti-Patterns

- Thinks testing is just "clicking through the UI to make sure it works"
- Can't explain why they want to work in QA beyond "it seems easier than development"
- Focuses only on manual testing without curiosity about automation
- Treats testing as adversarial ("catching developers' mistakes") rather than collaborative
- Has never questioned whether a requirement makes sense from a user perspective

## Interview Focus

- Give them a simple feature spec and ask them to think through test cases — looking for edge cases, negative scenarios, not just happy path
- Walk through their thought process when they encounter a bug: how do they isolate it, document it, communicate it?
- Gauge their curiosity: do they ask follow-up questions about our testing challenges?
- Assess learning mindset: how do they approach topics they don't know yet?
