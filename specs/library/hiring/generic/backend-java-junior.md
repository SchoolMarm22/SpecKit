# What We're Looking For

I'm looking for someone who's written enough Java to understand why Spring exists, but hasn't been doing it so long that they've forgotten what it's like to learn something new. The best junior engineers I've hired could explain their code to me without using buzzwords they didn't understand.

This role is perfect for someone 1-3 years into their career who wants to grow into larger system design challenges. You'll start owning small services and APIs, with plenty of mentorship as you learn how backend systems actually work at scale. I care more about your ability to debug a null pointer exception methodically than whether you've memorized every Spring annotation.

## Must-Haves

- Can write and test a REST API in Spring Boot from scratch
- Understands SQL joins and can write queries without an ORM holding your hand
- Has debugged production issues, even small ones (shows ownership mindset)
- Can explain the difference between checked and unchecked exceptions
- Basic understanding of HTTP status codes and when to use them

## Strong Signals

- Has contributed to a team codebase, not just personal projects
- Asks good questions during technical discussions (shows curiosity)
- Can read and understand existing code before asking for help
- Has set up local development environments without extensive hand-holding

## Anti-Patterns

- Only knows Java through Spring Boot tutorials (no fundamentals)
- Can't explain what their code does without reading it line by line
- Treats every problem as a framework problem rather than a logic problem
- Gets defensive when code is reviewed or questioned
- Has never worked with a database more complex than an in-memory H2

## Interview Focus

- Problem-solving approach: how do they break down unknown problems
- Code reading skills: can they understand and modify existing Java code
- Learning mindset: how do they approach concepts they don't know yet
- Debugging instincts: what's their process when something doesn't work
- Communication: can they explain their thinking clearly as they code
