# What We're Looking For

I'm looking for someone who genuinely gets excited about mobile development, not just someone who ended up here because "mobile pays well." The best junior iOS engineers I've hired had personal projects or were tinkering with SwiftUI before we even posted the job.

You don't need years of experience, but you do need to show you can learn independently. Mobile development moves fast — iOS releases yearly, Swift evolves constantly, and what worked in iOS 15 might be deprecated in iOS 17. I need someone who stays curious and doesn't get overwhelmed by change.

## Must-Haves

- Built and shipped at least one iOS app (App Store, TestFlight, or personal project)
- Comfortable with Swift fundamentals and can explain optionals, protocols, and closures
- Understanding of Auto Layout constraints and can build responsive UI without relying entirely on Storyboards
- Can debug using Xcode's debugging tools and knows how to read crash logs

## Strong Signals

- Has opinions about SwiftUI vs UIKit (either direction, but can articulate why)
- Contributed to open source or has public code repositories
- Understands memory management basics (ARC, retain cycles, weak references)
- Shows awareness of iOS design patterns (MVC, MVVM, delegation)

## Anti-Patterns

- Only ever used drag-and-drop interface builders without understanding the underlying code
- Can't explain the difference between a view and a view controller
- Treats Stack Overflow as documentation rather than understanding the underlying concepts
- Dismisses code reviews as "nitpicking" rather than learning opportunities
- Never tested their app on actual devices, only simulator

## Interview Focus

- Code walkthrough of their personal project — what decisions they made and why
- How they approach learning new iOS concepts when documentation isn't clear
- Debugging process when their app crashes or behaves unexpectedly
- What excites them most about iOS development and what they want to learn next
