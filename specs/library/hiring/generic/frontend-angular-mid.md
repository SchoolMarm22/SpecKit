# What We're Looking For

I need someone who's moved beyond the "just follow the ticket" phase but isn't quite ready to architect entire features yet. You should be comfortable working independently on medium-sized features and have opinions about code quality that go beyond "it works."

The sweet spot is someone who's hit their stride with Angular but still remembers what it was like to be confused by RxJS. You should be the person junior engineers come to with questions, and you should actually enjoy helping them figure things out.

## Must-Haves

- 2-3 years building production Angular applications (not just tutorials)
- Comfortable with RxJS beyond basic subscribe/unsubscribe patterns
- Has written unit tests that actually caught bugs, not just hit coverage targets
- Can implement responsive designs from Figma mockups without constant designer feedback
- Understands Angular lifecycle hooks and when to use each one

## Strong Signals

- Has migrated an Angular app between major versions (shows they deal with real codebases)
- Comfortable with NgRx or similar state management (bonus if they can explain when you don't need it)
- Has worked with design systems or component libraries
- Can spot performance issues in Angular apps and knows basic optimization techniques

## Anti-Patterns

- Only worked on greenfield projects with perfect setups
- Treats TypeScript like JavaScript with type annotations slapped on
- Can't explain why they chose a particular Angular pattern beyond "that's how the tutorial did it"
- Defensive about code review feedback or dismisses accessibility concerns
- Has never worked with legacy Angular code (everything we build today becomes legacy tomorrow)

## Interview Focus

- Code review scenarios: how they'd give feedback on junior engineer's PR
- Component design decisions: when to break things apart, when to keep them together
- Debugging approach when an Observable chain isn't working as expected
- How they balance feature velocity with technical debt
