# What We're Looking For

I need someone who's graduated from copying Stack Overflow answers to actually understanding why the code works. Mid-level means you can take a feature spec and own it end-to-end without constant hand-holding, but you're still learning system design fundamentals.

You should have real production Python experience — not just Django tutorials or Flask hello-world apps. I want to see someone who's debugged performance issues in production, dealt with database migrations that didn't go as planned, and has opinions about code structure that come from maintaining other people's code.

## Must-Haves

- 2-4 years production Python experience with Django, Flask, or FastAPI
- Built and maintained REST APIs that handle real user traffic
- Database design experience beyond basic CRUD (indexes, migrations, query optimization)
- Git workflow competency including code reviews and merge conflict resolution
- Has fixed production bugs under pressure

## Strong Signals

- Can explain the difference between sync and async Python and when to use each
- Experience with testing beyond just hitting 80% coverage (unit, integration, mocking strategies)
- Has worked with message queues or background job processing (Celery, RQ, etc.)
- Comfortable with Docker and basic deployment concepts
- Can articulate why they chose one Python framework over another

## Anti-Patterns

- Only has experience with toy projects or bootcamp assignments
- Can't explain their database schema choices or why they indexed certain columns
- Writes code that works but can't maintain or extend their own work from 6 months ago
- Dismisses code reviews as nitpicking rather than learning opportunities
- Has never dealt with API versioning or backwards compatibility

## Interview Focus

- Code structure and maintainability: show me how they organize a non-trivial Python service
- Database thinking: how they approach schema design and query performance
- Production debugging: walk through finding and fixing a real performance issue
- API design decisions: why they structure endpoints the way they do
