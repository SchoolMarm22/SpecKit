# What We're Looking For

I need someone who's moved beyond copying Stack Overflow snippets and actually understands the Java ecosystem. You should be able to pick up a moderately complex feature, break it down into sensible components, and ship it without constant hand-holding.

At the mid-level, I'm looking for engineers who have opinions about their craft. Not necessarily the *right* opinions, but thoughtful ones backed by experience. You should have been burned by a few architectural decisions and learned from them. Most importantly, you should be ready to own entire features end-to-end, not just implement tickets someone else designed.

## Must-Haves

- 3+ years of production Java experience with Spring Boot or equivalent framework
- Designed and implemented REST APIs that handle real user traffic (not just CRUD tutorials)
- Experience with relational databases and can write SQL beyond simple SELECT statements
- Has debugged production issues using logs, metrics, or profiling tools
- Can read and contribute meaningfully to code reviews

## Strong Signals

- Experience with microservices architecture and the tradeoffs involved
- Has worked with message queues (Kafka, RabbitMQ, SQS) in production
- Understands JVM tuning basics and garbage collection concepts
- Can articulate why they chose one data structure or algorithm over another
- Has mentored a junior engineer or led a small technical initiative

## Anti-Patterns

- Only ever worked on monoliths and thinks microservices solve everything
- Writes Java like it's C++ (or writes Java like it's JavaScript)
- Can't explain the difference between checked and unchecked exceptions
- Has never written a unit test that wasn't auto-generated
- Treats every performance problem as a database problem

## Interview Focus

- System design for a medium-complexity feature (not distributed systems, but more than a single endpoint)
- Java fundamentals: collections, concurrency, memory management
- How they approach debugging when something breaks in production
- Code review scenarios: what would they flag and why
- Past technical decisions they'd make differently today
