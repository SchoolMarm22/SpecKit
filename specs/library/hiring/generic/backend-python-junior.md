# What We're Looking For

I'm looking for someone early in their career who genuinely enjoys problem-solving and has shown they can learn independently. Junior doesn't mean we'll hold your hand through everything — you should be comfortable reading documentation, debugging your own code, and asking specific questions when you're stuck.

What matters more than years of experience is evidence that you can write code that works, think through edge cases, and communicate about technical decisions. I've hired bootcamp grads who outperformed CS majors because they had built real projects and could explain their choices.

The best junior engineers I've worked with are curious about how systems work under the hood and aren't satisfied with just copying Stack Overflow answers without understanding them.

## Must-Haves

- Built and deployed at least one Python web application (Flask, Django, or FastAPI)
- Comfortable with SQL and can write queries beyond basic SELECT statements
- Has used Git for more than just personal projects (ideally with pull requests and code reviews)
- Can explain the difference between a list and a dictionary and when you'd use each
- Understands HTTP basics (status codes, headers, GET vs POST)

## Strong Signals

- Has contributed to open source, even small documentation fixes
- Built something with an API integration (third-party service, not just internal endpoints)
- Can walk through their code and explain design decisions they made
- Asks follow-up questions during technical discussions
- Has used testing frameworks like pytest, even if not test-driven development

## Anti-Patterns

- Only has tutorial-level experience (todo apps, following along with courses)
- Can't explain why they chose Python over other languages for their projects
- Treats errors as mysterious black boxes instead of debugging systematically
- Expects detailed requirements for every task without trying to understand the bigger picture
- Dismisses code reviews as unnecessary bureaucracy

## Interview Focus

- Code walkthrough of a project they built: what problems did they solve and how?
- Debugging approach: how do they investigate when something isn't working?
- Learning examples: tell me about a time you had to figure out something new
- Growth mindset: what technical skills are they excited to develop next?
