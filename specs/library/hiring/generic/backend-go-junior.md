# What We're Looking For

I'm looking for someone who's written real Go code beyond tutorials — even if it's just a personal project or bootcamp work. The key is they chose Go deliberately and can explain why they liked it over Python or Node.js they might have used before.

Don't need someone who's architected microservices at scale, but I do need someone who understands that code needs to be maintained by other people. If they've only ever written code for themselves, this won't work. Looking for signs they think about code as communication, not just instructions for computers.

We have a strong pairing culture and invest heavily in mentorship. The right person will be curious about system design even if they can't design systems yet, and will ask good questions during code reviews rather than just nodding along.

## Must-Haves

- Has written a complete Go application (even if small) with proper error handling and logging
- Understands HTTP fundamentals — can explain the difference between PUT and POST, knows what status codes mean
- Has worked with at least one database (SQL or NoSQL) and written queries by hand, not just through an ORM
- Can explain goroutines and channels at a basic level, even if they haven't built concurrent systems
- Has used Git in a collaborative setting (school project, open source, previous job)

## Strong Signals

- Contributed to open source projects, even just documentation fixes
- Has opinions about code formatting and can explain why consistency matters
- Built something with tests, even if the test coverage isn't perfect
- Can walk through their own code and point out parts they'd improve

## Anti-Patterns

- Only ever copied code without understanding what it does
- Thinks "it works on my machine" is an acceptable answer to deployment questions
- Has never read anyone else's code or worked on a team
- Treats error handling as optional or "something to add later"
- Can't explain any tradeoffs in their technical decisions

## Interview Focus

- Code reading skills — give them Go code to read and explain, not write
- How they approach debugging when something doesn't work
- What they do when they encounter unfamiliar concepts or libraries
- Whether they can take feedback on their code without getting defensive
