# What We're Looking For

I care more about how someone thinks through problems than their years of experience. A junior engineer who can articulate why they chose useState over useReducer in a specific context is more valuable than someone who's memorized every React hook but can't explain when to use them.

Looking for someone who's built something real — doesn't have to be at a company, could be a side project or bootcamp capstone. But they need to have wrestled with actual problems: API calls that fail, components that re-render too much, CSS that breaks on mobile. Book knowledge only goes so far.

The best junior engineers I've hired were curious about the "why" behind the patterns, not just copying code from Stack Overflow. They ask good questions during code reviews and actually incorporate feedback instead of getting defensive.

## Must-Haves

- Built at least one complete React application with multiple components and state management
- Can explain the difference between props and state without just reciting definitions
- Has written CSS that works across different screen sizes (doesn't need to be perfect, but shows awareness)
- Comfortable with Git basics: branching, merging, handling merge conflicts
- Can read and understand existing JavaScript/TypeScript code, even if they didn't write it

## Strong Signals

- Has opinions about code organization, even if they're still forming
- Asks clarifying questions about requirements instead of just implementing what they think was asked
- Shows examples of iterating on their own work — version 1 vs version 2 of a project
- Has debugged something that wasn't working and can walk through their process

## Anti-Patterns

- Can only work with tutorial-perfect code; gets lost when things are messy
- Dismisses CSS as "not real programming" or always reaches for a library first
- Never asks questions during technical discussions (either too scared or thinks they know everything)
- Has only worked on solo projects and can't talk about collaborating with others
- Expects every technical decision to have one "right" answer

## Interview Focus

- Problem-solving process: give them a broken component and see how they debug it
- Communication: can they explain their code to someone who didn't write it?
- Learning mindset: ask about something they struggled with and how they worked through it
- Code review simulation: show them code with issues and see what they notice
