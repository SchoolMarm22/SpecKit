# What We're Looking For

I'm looking for someone who's hungry to learn and has already shown they can turn ideas into working code. This isn't an internship — you'll be shipping real features to real users within your first month. But I care more about your potential trajectory than your current skill ceiling.

The best junior engineers I've hired were the ones who could explain their thinking process, not just their final solution. They'd built something — anything — outside of bootcamp assignments and could walk me through the decisions they made along the way.

## Must-Haves

- Built at least one complete Angular application (personal project, bootcamp capstone, or internship work counts)
- Understands TypeScript fundamentals — can explain the difference between `any` and `unknown`
- Has used Angular CLI and understands the component lifecycle (ngOnInit, ngOnDestroy matter)
- Can write basic RxJS — you don't need to be an expert, but you should know what an Observable is
- Demonstrates curiosity about web fundamentals (can explain what happens when you type a URL)

## Strong Signals

- Has contributed to open source, even if it's just documentation fixes
- Asks good questions in technical discussions (shows they're thinking, not just absorbing)
- Has built something with state management (NgRx is nice, but even just managing component state thoughtfully counts)
- Shows interest in testing — doesn't need to be TDD expert, but has written at least basic unit tests

## Anti-Patterns

- Only knows Angular from tutorials, never built anything from scratch
- Can't explain why they chose Angular over React (shows they haven't thought about tradeoffs)
- Treats TypeScript like JavaScript with extra syntax (misses the point of static typing)
- Never looks at browser dev tools or doesn't know how to debug their own code
- Expects someone else to make all architectural decisions for them

## Interview Focus

- Walk through a personal project: What problems did you encounter and how did you solve them?
- Basic Angular concepts: Can you explain dependency injection without just reciting definitions?
- Problem-solving approach: How do you debug when something isn't working?
- Growth mindset: What's something you learned recently that changed how you think about frontend development?
