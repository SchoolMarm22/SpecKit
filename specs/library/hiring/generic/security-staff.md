# What We're Looking For

I need someone who thinks about security as a system-level property, not a checklist of tools. Staff engineers at this level should be able to look at our entire technology stack and identify the three most critical risks we're not addressing — and then build consensus around fixing them.

This isn't a penetration testing role or a compliance checkbox role. This is for someone who can architect security into systems from the ground up, influence engineering teams across the org, and make technical decisions that affect how we build software for the next two years. You should have strong opinions about threat modeling, defense in depth, and what "shift left" actually means in practice.

## Must-Haves

- Designed and implemented security architecture for distributed systems at scale (1M+ users)
- Led cross-functional security initiatives that required buy-in from multiple engineering teams
- Built or significantly improved an organization's threat modeling process
- Hands-on experience with both application security and infrastructure security (not just one or the other)
- Can code well enough to review security-critical pull requests and spot real vulnerabilities

## Strong Signals

- Has strong opinions about SAST/DAST tools based on actual experience running them in production
- Experience building security tooling that developers actually want to use
- Can articulate when zero-trust is worth the complexity cost and when it isn't
- Has been in an incident where a security assumption turned out to be wrong

## Anti-Patterns

- Security experience that's primarily compliance-driven rather than risk-driven
- Can't explain security tradeoffs in terms that product managers understand
- Treats security as something that happens after development rather than during
- "Just use [insert security vendor]" mentality without understanding the underlying problems
- Never worked somewhere that had a real security incident

## Interview Focus

- System design with security constraints: how do you secure a system while keeping it maintainable?
- Influence without authority: how do you get engineering teams to adopt security practices?
- Risk prioritization: if you could only fix three things in our stack, what would they be and why?
- Technical depth: can they spot the security implications of architectural decisions in real-time?
