# What We're Looking For

I need someone who has grown with Vue through its evolution — from Vue 2 to Vue 3, who understands why the Composition API exists and when *not* to use it. Generic "React developer who learned Vue last month" won't cut it here.

We're looking for architectural judgment. Someone who's made the hard calls about state management (Vuex vs Pinia vs composables), who's debugged reactivity issues in production, and who can mentor our L3s through the subtleties of Vue's reactivity system. You should have opinions about component composition patterns and be able to defend them.

This role owns our component library and sets frontend standards across three product teams. You'll be the person other engineers come to when they're stuck on complex state interactions or performance bottlenecks.

## Must-Haves

- 4+ years with Vue in production, including migration experience (Vue 2 to 3 or major version upgrades)
- Built and maintained a component library or design system used by multiple teams
- Deep understanding of Vue's reactivity system and can debug performance issues with Vue DevTools
- Experience with modern Vue ecosystem: Vite, Composition API, Pinia, and TypeScript integration
- Has mentored junior engineers through Vue-specific challenges (not just generic JavaScript)

## Strong Signals

- Contributed to Vue ecosystem packages or has published Vue-related tooling
- Can articulate the tradeoffs between Options API and Composition API for different use cases
- Experience with Vue SSR/Nuxt in production environments
- Has strong opinions about Vue component testing strategies (Vue Test Utils, Cypress, etc.)
- Led technical discussions about frontend architecture decisions

## Anti-Patterns

- Treats Vue like React with different syntax (doesn't understand Vue's philosophy)
- Can't explain when to use refs vs reactive vs computed
- Only experience is with Vue 3 — hasn't lived through the ecosystem evolution
- Dismisses Vue DevTools or doesn't use them for debugging
- "Senior" experience that was really just implementing designs from Figma

## Interview Focus

- Vue-specific architectural decisions: when they chose Vuex vs Pinia vs local state
- Real performance debugging stories using Vue DevTools
- How they've handled reactivity gotchas in complex applications
- Component API design philosophy and how they balance flexibility vs consistency
- Mentoring examples specific to Vue concepts, not generic coding principles
